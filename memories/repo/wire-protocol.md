# Wire Protocol

Authoritative contract between the C++ backend and the React frontend.
The user-facing version lives in `README.md` → "Wire protocol"; this
file is the internal memory that records the *why* and the invariants
future agents must preserve.

## Transport
- WebSocket endpoint `/*` on port 9090 (prod), 9098 (test backend), 9099 (playwright mock).
- Framing: JSON-RPC 2.0 (requests carry `id`, notifications omit it).
- Server → Client pushes are notifications (no `id`).

## RPC methods

### Requests (client expects a reply)
- `vendor.getConfigSchema { vendor }` → `{ fields: [...] }`
- `nodes.list {}` → `{ nodes: [{ type, label, category, ports: [{ id, direction, dataType }] }] }` sorted by `type`. Source of truth for what this backend build can execute.
- `workflow.execute { nodes, edges, breakpoints }` → `{ status: "started", run_id }`. Validates params shape; failures throw `InvalidParams` → `-32602`.
- `workflow.cancel {}` → `{ cancelled: true, run_id }`. Semantically identical to the legacy `workflow.stop` notification but returns the id it interrupted. Mid-node preemption is NOT implemented — cancel is observed at the next node boundary.
- `debug.add_breakpoint { node_id }` → `{ ok: true }`
- `debug.remove_breakpoint { node_id }` → `{ ok: true }`

### Notifications (no reply)
- `workflow.stop {}` — legacy alias of `workflow.cancel`; kept for backward compat.
- `debug.continue {}`
- `debug.step_over {}`

### Server → Client pushes
- `node.status { node_id, status, run_id?, elapsed_ms?, output?, error?, kind?, reason?, upstream?, runs_count?, avg_ms? }`
  - `status`: `idle|running|done|error|paused|skipped`
  - `kind`: `missing_input|invalid_config|runtime|upstream_failed`
  - `reason` (status=`skipped`): `branch_pruned|upstream_failed`
- `workflow.complete { run_id? }`
- `debug.paused { node_id, run_id?, inputs: [{ handle, source, value: {type, ...} }] }`
  - Tensor preview values are truncated to ≤ 16 floats.

## run_id contract (R1)
- Format: `run-<seq>-<ms>` (process-local counter + steady_clock ms since process start).
- Generated by `RunSession::start` AFTER joining the prior worker, so no stale event can be mis-tagged.
- Stamped onto every `node.status` / `debug.paused` event by `Executor::notify_status` and the pause path.
- Frontend `WorkflowRunner::setActiveRunId` stores the id returned by `workflow.execute`; events with a mismatched `run_id` are silently dropped. Events *without* `run_id` (legacy backend, tests) are accepted verbatim — the filter is strictly additive.

## Graph validation (S1)
- Runs at the start of `Executor::execute`, *after* the RPC boundary.
- On failure emits a single `__workflow__` / `validation_failed` push with an `errors[]` array, then `workflow.complete`. No per-node errors.
- Error kinds: `unknown_node_type | dangling_edge | unknown_port | type_mismatch`.
- Type compatibility lattice: identical ≡, `generic` wildcard, `image→tensor` implicit coercion, branch-source-asymmetric (branch source → any non-branch target; branch target ← branch source only). Mirrored in `frontend/src/nodes/portSchema.ts`.

## JSON-RPC error codes
| Code | When |
|------|------|
| -32700 | Parse error (malformed JSON) |
| -32601 | Method not found |
| -32602 | Invalid params — shape-level validation at RPC boundary, caller bug |
| -32000 | Server error — any other handler exception |

Rule: use -32602 only for shape problems diagnosable without the
handler catalog. Graph-level problems go through the
`validation_failed` push instead.

## Invariants that must NOT regress
1. Every server→client push method name already in use must keep its
   current shape; new fields are allowed only if optional.
2. `workflow.execute` reply must include `run_id` (added in R1).
3. `workflow.stop` must remain a notification, not a request (breaks
   every older client if flipped).
4. `run_id` format is opaque to clients — never parse it as numbers,
   only compare as strings.
5. Untagged events must still be accepted by the FE filter.
